**Change B — Insert foil_power field into TelemetryPacket (lines 99-107):**

Replace the existing struct with:
```c
//Telemetry to send, MUST BE 8-bit!!
struct __attribute__((packed)) TelemetryPacket {
    uint8_t foil_bat = 0xFF;    // index 0 — battery % 0-100
    uint8_t foil_temp = 0xFF;   // index 1 — FET temp degC
    uint8_t foil_speed = 0xFF;  // index 2 — speed km/h
    uint8_t error_code = 0;     // index 3 — fault flags
    // Phase 1: SCALE=watts/50  DECODE(Tx Display.ino)=foil_power*50  range 0-12750W at 50W resolution
    uint8_t foil_power = 0xFF;  // index 4 — power (watts/50); 0xFF = not available
    // This must be the last entry
    uint8_t link_quality = 0;   // index 5 (was 4)
} telemetry;
```

CRITICAL ordering rules:
- foil_power MUST go between error_code (index 3) and link_quality. Do NOT insert before error_code.
- link_quality MUST remain the last entry (the comment says so; Radio.ino depends on it staying last for the wrap).
- Do not change any default values for existing fields.

**Backwards compatibility note (no code change needed, just awareness):**
Old Tx firmware has sizeof(TelemetryPacket)==5. When new Rx sends foil_power at index 4, old Tx bounds check `4 < 5` is TRUE — it writes the foil_power byte into its link_quality slot. This causes the old Tx link quality display to show power-derived garbage one cycle in five. This is cosmetic only, not a safety issue, and is the documented expected behavior. No workaround needed.

  <action>
One addition in VESC.ino — inside the `#ifdef VESC_MORE_VALUES` block in `getValuesSelective()`.

Locate the existing block (currently lines 66-70):
```c
#ifdef VESC_MORE_VALUES
    motCur = buffer_get_int32(message, &cnt);
    batCur = buffer_get_int32(message, &cnt);

```

Add the power calculation AFTER the `duty` parse line and BEFORE the `#endif`. The final block must read:
```c
#ifdef VESC_MORE_VALUES
    motCur = buffer_get_int32(message, &cnt);
    batCur = buffer_get_int32(message, &cnt);
    duty   = buffer_get_int16(message, &cnt);

    // Power calculation
    // batCur unit: int32 in mA*100. Divide by 100000.0f to get Amps.
    float batCur_amps = (float)batCur / 100000.0f;
    // fbatVolt is global float set at end of this function from previous cycle (one-cycle lag, acceptable)
    float watts = fbatVolt * batCur_amps;
    if (watts < 0.0f) watts = 0.0f;  // Clamp: ignore regenerative braking (negative current)
    // SCALE: foil_power = watts/50  |  DECODE on Tx Display.ino: watts = foil_power * 50  (range 0-12750W)
    telemetry.foil_power = (uint8_t)constrain(watts / 50.0f, 0.0f, 255.0f);

    #ifdef DEBUG_VESC
    Serial.print("V="); Serial.print(fbatVolt);
    Serial.print(" I="); Serial.print(batCur_amps);
    Serial.print(" W="); Serial.print(watts);
    Serial.print(" encoded="); Serial.println(telemetry.foil_power);
    #endif
#endif
```

Do NOT change anything outside this block. Do NOT change VESC_PACK_LEN (Phase 1 adds no new VESC bitmask bits — the existing bitmask already requests batCur). Do NOT reorder the parse calls for batVolt (fbatVolt from previous cycle is correct per Option A in RESEARCH.md).

**Unit error guard:** The divisor MUST be `100000.0f` (one hundred thousand). Using `/100` or `/1000` causes 100x or 10x power inflation causing immediate saturation to 255. The comment is mandatory.

**Regen guard:** The `if (watts < 0.0f) watts = 0.0f;` line is mandatory. Without it, negative float cast to uint8_t is undefined behavior in C (may wrap to 255 or crash).
